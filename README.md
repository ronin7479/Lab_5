#Контрольні питання
1. Основи Git
Яка різниця між Git та GitHub?

Git - це локальна розподілена система контролю версій.

GitHub - це хостинг Git-репозиторіїв та платформа для співпраці (pull requests, issues, CI тощо).

Що означає "розподілена" система контролю версій?

Кожен розробник має повну копію репозиторію з історією. Для більшості операцій інтернет не потрібен.

Яка різниця між git init та git clone?

git init - створює новий порожній Git-репозиторій у поточній папці.

git clone - копіює існуючий віддалений репозиторій разом з історією.
2. Staging area та commits
Що таке Staging Area і навіщо вона потрібна?
Staging Area - місце, де ви формуєте набір змін перед комітом. Це дозволяє вибірково додавати файли та контролювати, що саме потрапить у commit.
Яка різниця між git add . та git add -A?
git add . - додає нові та змінені файли, але не завжди враховує видалення.
git add -A - додає всі зміни, включаючи видалення.
Як написати гарне commit message?
Короткий заголовок 
В наказовому стилі: “Fix bug”, “Add feature”
Далі - деталі, якщо потрібно
Має пояснювати, чому зроблено зміну, а не лише що
Як виправити останній коміт?
git commit --amend
Дозволяє змінити повідомлення або додати пропущені файли.

3. Гілки та злиття
Що таке гілка в Git і чому вони "легковажні"?
Гілка - це просто вказівник на певний коміт.
“Легковажні”, бо створюються/видаляються дуже швидко й займають мінімум пам’яті - це лише покажчики.
Яка різниця між git merge та git rebase?
merge - створює новий коміт злиття, історія розгалужується.
rebase - переписує історію, ставить коміти поверх іншої гілки, робить історію лінійною.
Що таке fast-forward merge?
Якщо гілка не має власних комітів після відгалуження, merge просто пересуває вказівник на новіший коміт - без створення merge-коміта.
Коли виникають merge конфлікти і як їх уникнути?
Коли дві гілки змінюють одні й ті самі рядки в одному файлі.
Уникати можна, якщо:
частіше робити pull/rebase,
розділяти роботу на менші частини,
уникати великих “монолітних” змін.
Чому не можна робити rebase публічних комітів?
Rebase переписує історію, і всі інші учасники, що вже скачали стару версію, отримають конфлікти та проблеми з синхронізацією.

4. Віддалені репозиторії
Що таке origin?
Стандартна назва для основного віддаленого репозиторію, з якого було зроблено clone.
Яка різниця між git fetch та git pull?
fetch - отримує нові коміти, але не зливає їх у вашу гілку.
pull = fetch + merge.
Що робить git push --force і чому це небезпечно?
Перезаписує історію у віддаленій гілці.
Небезпека: можна видалити чужі коміти або зламати історію для інших.
Як видалити віддалену гілку?
git push origin --delete my-branch

5. Скасування змін
Яка різниця між git reset --soft, --mixed та --hard?
soft - відкат комітів, зміни лишаються в staging.
mixed - відкат комітів, зміни переходять у робочу директорію.
hard - відкат комітів та видалення змін з файлів.
Яка різниця між git reset та git revert?
reset переписує історію.
revert створює новий коміт, що відміняє зміни — історія залишається чистою.
Як скасувати git add (видалити файл зі staging)?
git reset HEAD <file>
Як повернути видалений файл?
git checkout HEAD -- <file>
Або сучасний варіант:
git restore <file>

6. Безпека
Що не повинно потрапляти в Git репозиторій?
Паролі, токени, ключі

Конфігурації з секретами
Великі файли
Згенеровані файли
Що робити якщо випадково закомітили пароль?
Видалити з файлів
Змінити пароль/токен
Переписати історію, якщо треба повністю стерти з історії.
HTTPS vs SSH для GitHub - що краще?
HTTPS - простіше, працює всюди, але потребує PAT або пароль.
SSH - безпечніше та зручніше для регулярної роботи (ключі).
Що таке Personal Access Token (PAT)?
Токен, що замінює пароль GitHub у HTTPS-операціях. Дає контрольований доступ із правами, які ви визначаєте.

7. GitHub
Що таке Pull Request?
Запит на перегляд та злиття змін у іншу гілку. Основний механізм код-рев’ю.
Яка різниця між Issues та Pull Requests?
Issues - задачі, баги, обговорення.
PR - пропозиції змін у коді.
Що таке Fork і навіщо він потрібен?
Повна копія чужого репозиторію у вашому акаунті GitHub. Використовується для внесення змін у проєкти без доступу до їхніх репозиторіїв.
Що таке GitHub Actions?
CI/CD-платформа від GitHub для автоматизації: тести, білди, деплой, перевірки.

8. Best Practices
Як часто треба робити коміти?
Часто, маленькими логічними частинами
Кожен коміт має бути робочим і зрозумілим
Скільки часу має жити feature branch?
Якнайкоротше - ідеально кілька днів, максимум 1–2 тижні. Довгі гілки = конфлікти.
Як організувати git workflow в команді?
GitHub Flow простий, 1 гілка main + feature branches
Git Flow release/hotfix/feature, складніший
Trunk-based development часті маленькі коміти в main
Чи треба комітити node_modules або інші залежності?
Ні.
Вони великі, генеруються автоматично і повинні бути у .gitignore.
